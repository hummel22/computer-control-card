# AGENTS.md

## Project overview

`computer-control-card` is a frontend-only Home Assistant Lovelace custom card for controlling and monitoring computers remotely. The card is implemented as a Lit-based web component and includes a standalone Vite demo used for local development, end-to-end checks, and screenshot capture without requiring a live Home Assistant instance.

Key areas:

- `src/` contains the card source code, state helpers, action handling, styles, and shared types.
- `demo/` contains the standalone browser demo, fixtures, styles, and mocked Home Assistant object.
- `tests/unit/` contains Node-based unit tests.
- `tests/e2e/` contains Playwright tests that exercise the standalone demo and capture screenshots.
- `dist/` is the build output generated by Vite.
- `test-results/` and `artifacts/` contain Playwright outputs, reports, screenshots, traces, and related test artifacts.

## Setup

Use npm with the checked-in lockfile:

```sh
corepack enable || true
npm ci
npx playwright install --with-deps chromium
```

## Build commands

```sh
npm run build
```

For local development:

```sh
npm run dev
npm run demo
```

## Typecheck, lint, and test commands

```sh
npm run typecheck
npm run lint
npm test
npm run test:e2e
```

Notes:

- `npm run lint` currently runs the TypeScript compiler with `--noEmit`.
- `npm test` runs unit tests in `tests/unit/*.test.ts` through `tsx --test`.
- `npm run test:e2e` runs all Playwright tests against the standalone demo.

## Visual test commands

Run the standalone demo screenshot tests after any UI-facing change:

```sh
npm run screenshots
```

Equivalent command:

```sh
npm run test:visual
```

These commands run the Playwright tests tagged with `@screenshots` and write screenshot artifacts for review.

## Expected output locations

- Production build output: `dist/`.
- Playwright test output, traces, and failure artifacts: `test-results/`.
- Playwright HTML report: `artifacts/playwright-report/`.
- Standalone demo screenshot artifacts: `artifacts/screenshots/`.

## Required testing rules

- After UI changes, always run the standalone demo screenshot tests with `npm run screenshots`.
- If screenshots change intentionally, update the visual snapshots or screenshot artifacts as appropriate and mention the intentional screenshot update in the final response.
- Tests must never require access to a real Home Assistant instance.
- Do not use secrets or real network devices in development, tests, fixtures, screenshots, or documentation examples.
- Service calls in tests must use mocked `hass.callService`; assert the mocked calls instead of sending real Home Assistant service calls.
